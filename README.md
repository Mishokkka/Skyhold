# FBL Skyhold Manager

Личный модуль менеджмента владений для Forbidden Lands в Foundry VTT.

Поддерживаемая среда: Foundry VTT `13.351`, система `forbidden-lands` `13.0.5`.

Модуль предназначен для ведения поселений, кораблей, аванпостов, организаций и островов-летяг. Главная задача: держать жителей, здания, склад, экономику, оборону и события владения в одном окне, без ручных таблиц рядом с Foundry.

## Установка

1. Распакуй модуль в папку `Data/modules/fbl-skyhold`.
2. Проверь, что файл `module.json` лежит прямо внутри `fbl-skyhold`, а не во вложенной папке.
3. Включи модуль в мире Foundry.
4. После загрузки мира используй кнопку модуля или API `game.fblSkyhold.open()`.

## Что умеет модуль

Модуль хранит несколько владений одновременно. Тип владения может быть поселением, кораблем, аванпостом, организацией или островом-летягой.

Для каждого владения есть вкладки:

- обзор;
- жители;
- здания;
- оборона;
- особое;
- склад;
- GM.

Игроки могут видеть публичные владения. Скрытые GM-владения не должны открываться через обычные окна, редакторы и сценические отметки.

## Обзор владения

В обзоре хранятся статус, период, публичное описание, заметки, население, мораль, репутация, Defence Rating, погода, высота, дрейф, механизмы и развитие.

Развитие делится на четыре направления:

- еда;
- технология;
- культура / религия;
- война.

Здания могут добавлять развитие. Жители напрямую развитие не повышают, они работают через здания, бригады, оборону и производство.

## Жители

Житель хранит имя, народ, пол, возраст, культуру, веру, социальное прошлое, черты, работу, жилье, зарплату, мораль, ранения и статус смерти.

Возраст нормализуется по народу. Используются категории вроде молодой, взрослый и старый, но возраст хранится числом.

Жители могут быть назначены:

- в здания;
- в жилье;
- в строительные бригады;
- в оборонные отряды;
- командиром обороны.

При удалении жителя модуль чистит ссылки на него в зданиях, бригадах и обороне.

## Здания

Здания бывают обычными объектами владения, а не только шаблонами. Шаблон используется как заготовка, после постройки объект живет отдельно и может быть отредактирован.

Здание может иметь функции:

- производство;
- доход;
- оборона;
- жилье;
- склад;
- культура / религия.

Поддерживаются рабочие места, подходящие типы работников, эффективность, строительство, ремонт, повреждения, содержание, материалы, требования по развитию, религиозные настройки, производственные линии и внутренние складские помещения.

Лечебные здания получают медицинский блок, но обычная загрузочная нормализация больше не должна стирать пользовательские функции здания. Жесткое восстановление медицинского профиля вынесено в repair-режим.

## Строительство

Строительство использует материалы, сложность, цель успехов, прогресс и строительные бригады.

Бригады могут иметь участников, лидера, подходящие типы работников и временное распределение по объектам.

Если здание или житель удаляется, связанные назначения чистятся, чтобы в данных не оставались мертвые ID.

## Склад

Склад хранит ресурсы, предметы, деньги и лог операций.

Игроки берут и отдают вещи только через своего дефолтного персонажа. Модуль использует `game.user.character`, а если он не назначен, пробует выбранный токен как fallback.

Выбор актера в складском интерфейсе намеренно отсутствует. Это снижает шум и не раскрывает игрокам лишних актеров мира.

Склад поддерживает:

- ресурсы с количеством;
- предметы с сохранением item data;
- складские комнаты;
- отметку “на продажу”;
- казну в медных монетах с отображением в ЗМ / СМ / ММ;
- лог операций.

## Экономика

Экономика считает производство, доход, содержание и торговлю.

Производственные линии могут использовать работников, время, содержимое здания, расходы, накопление готового товара и ручной сбор.

Торговые здания могут продавать ресурсы и предметы со склада по бюджетам покупателей и эффективности цены.

## Оборона

Вкладка обороны хранит командира, подготовку ополчения, снаряжение, заметки, звуки исходов боя и оборонные отряды.

Отряд имеет сержанта / первый слот, четырех дополнительных участников, тип, экипировку и заметки.

Defence Rating используется как реальная механическая величина для защиты владения.

## Битва за владение

Массовая битва вынесена в отдельное окно. Она использует данные владения, оборонные отряды, здания, теги войск, подготовку, укрепления, планы сторон, угрозу раунда и масштаб столкновения.

Модуль отслеживает:

- силы защитников и нападающих;
- командование;
- позицию;
- стратегию;
- теги войск;
- брешь;
- прогресс цели;
- потери;
- урон владению;
- журнал раундов.

## Calendaria

Модуль умеет читать текущую дату из Calendaria, если интеграция доступна.

Хранятся последняя обработанная дата, текст даты, timestamp и количество прошедших quarter days. Это используется для ходов, производства и отчетов.

## Сценические отметки зданий

Здания можно перетаскивать на сцену как отметки. Отметка связывается со зданием и позволяет быстро открыть карточку объекта.

Иконка и изображение могут синхронизироваться с данными здания.

## Права игроков

Глобально ГМ решает, могут ли игроки открывать окно владений.

Для каждого владения отдельно настраивается, могут ли игроки:

- пользоваться складом;
- редактировать обзор;
- редактировать здания;
- редактировать жителей;
- редактировать оборону;
- редактировать подготовку битвы;
- редактировать особые записи.

Игроки не пишут напрямую в world settings. Клиент игрока отправляет patch-запрос ГМу, а клиент ГМа валидирует действие и применяет его к текущим данным.

Patch-запросы классифицируются по типу действия: `overview`, `storage`, `buildings`, `residents`, `defense`, `battle`, `special`. Смешанные запросы отклоняются, кроме разрешенных служебных побочных эффектов, например переназначения жителей при работе со зданиями или обороной.

## Хранилище данных

Основные данные разделены на три world setting:

- `skyholdMeta`;
- `skyholdHoldings`;
- `skyholdCatalog`.

Старый общий setting `skyholdData` сохраняется как rollback snapshot.

При записи используется commit marker. Модуль сначала помечает запись как начатую, затем сохраняет holdings и catalog, затем помечает meta как завершенную. Если загрузка обнаружит незавершенный или поврежденный commit, она попытается использовать rollback snapshot.

## Миграции, нормализация и repair

В модуле разделены три режима работы с данными.

`migration` переводит старую структуру данных в актуальную.

`normalization` безопасно доводит данные до валидного состояния при загрузке и обычной записи. Она не должна уничтожать пользовательские настройки.

`repair` предназначен для ручного жесткого восстановления, когда нужно вернуть объект к ожидаемой механической форме. Этот режим может быть разрушительным для пользовательских отклонений.

## Dev-проверка

В архив включен простой dev-check:

```bash
node scripts/dev/check.mjs
```

Он проверяет:

- синтаксис всех JS-файлов;
- циклические импорты;
- идемпотентность нормализаторов;
- мягкую нормализацию лечебных зданий;
- отклонение смешанных player patch;
- валидность commit marker;
- миграцию старого mock-сейва.

## API

После загрузки мира доступны:

```js
game.fblSkyhold.open();
game.fblSkyhold.rerender();
game.fblSkyhold.diagnostic();
```

## Ограничения

Модуль рассчитан на доверенный стол и авторский мир. Он не претендует на универсальный публичный менеджер поселений для всех кампаний Forbidden Lands.

Живая проверка в Foundry после обновления все равно нужна. Статическая проверка и dev-check ловят синтаксис, циклы и чистую логику, но не все особенности ApplicationV2, хуков, CSS-скоупинга и поведения конкретного мира.

## Разработка и CSS

CSS разделен на индексные файлы и узкие части. Карта структуры лежит в `styles/README.md`.

Главные команды проверки:

```bash
node scripts/dev/check.mjs
node scripts/dev/css-audit.mjs
```

`check.mjs` проверяет JS-синтаксис, циклы импортов, CSS-импорты, баланс фигурных скобок, лимит размера CSS-файлов, лимит `!important`, нормализаторы, player-write guard и commit marker.

`css-audit.mjs` показывает оставшиеся `!important`, повторяющиеся селекторы, точные дубликаты правил и классы, которые могут быть неиспользуемыми.


После CSS-унификации общий набор helper-классов описан в `styles/README.md`. Для новых экранов сначала используй `fbls-toolbar`, `fbls-action-group`, `fbls-card-header`, `fbls-chip` и `fbls-field`, а уже потом добавляй локальные паттерны.
